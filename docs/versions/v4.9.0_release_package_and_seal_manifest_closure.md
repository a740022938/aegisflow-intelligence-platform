## Seal Release Flow

```
approved → [seal-release] → sealed
                              ↓
                         releases record created
                         release_manifest.json generated
                         release_notes.md generated
                         3 audit events logged
                         model.promotion_status → sealed
                         model.release_id → set
```

### Release Manifest Contents
- Release name, version, sealed_at/by
- Source artifact details (id, name, type, model_family, framework, format, version, path)
- Source evaluation/experiment/dataset IDs
- Source model ID
- Metrics snapshot
- Approval record (id, status, approved_by, approved_at)
- Lineage chain (evaluation → artifact → model → approval)

### Release Notes Format
Markdown with:
- Version, sealed_at/by
- Artifact info
- Source evaluation/model/approval links
- Metrics section

### Safety Checks
- Only `promotion_status === 'approved'` artifacts can be sealed
- Double-seal prevented (checks `release_id`)
- Non-approved seal attempt returns 400

## Audit Events Generated per Seal

1. `seal_release` — category: release, target: artifact_id
2. `manifest_created` — category: release, target: release_id
3. `release_notes_created` — category: release, target: release_id
