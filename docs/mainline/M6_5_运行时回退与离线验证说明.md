# M6.5 运行时回退与离线验证说明

## 背景定性
- M6 方向正确：vision-bus catalog 插件化（发现层）是正确切入点。
- M6.5 失败点：运行时接入姿势错误，把插件初始化放到了请求链路。

## 本次处理原则
1. 先恢复主程序稳定。
2. 保留 M6 已完成成果，不删除发现层设计。
3. 插件系统先离线验证，再重接线上。

## 已执行动作
### 1) 运行时回退（主程序优先）
- 关闭 `local-api` 中插件系统默认启用。
- 禁止插件状态查询路由触发初始化。
- `vision-bus` 保留插件化代码，但默认关闭运行时发现，catalog 走 builtin fallback。

### 2) 离线验证（与 local-api 同环境）
- 新增独立脚本：`apps/local-api/scripts/m6_5_offline_plugin_validation.mjs`。
- 固定在 `apps/local-api` 目录执行，连续 20 次验证：
  - `import @agi-factory/plugin-runtime`
  - `new PluginManager()`
  - `listPlugins({ capability: 'vision' })`
- 结果：20/20 成功，无 `ERR_MODULE_NOT_FOUND`。

### 3) 模块边界修复（Node 运行时解析一致性）
- 对 `plugin-sdk` / `plugin-runtime` 做 NodeNext 对齐：
  - `tsconfig` 从 `ES2020 + bundler` 调整为 `NodeNext`。
  - 相对导入统一补齐 `.js` 后缀。
- 已重新构建：
  - `packages/plugin-sdk`
  - `packages/plugin-runtime`

## 当前状态
- `GET /api/health` 正常。
- `GET /api/vision/catalog` 正常，且为 builtin fallback（`plugin_system_active=false`）。
- 普通请求不再触发 PluginManager 初始化。

## 下一次正确重接（执行框架）
1. 启动期单次初始化（Fastify `onReady`）。
2. 初始化失败写入熔断缓存（避免请求期重复初始化）。
