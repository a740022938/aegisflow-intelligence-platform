# M16 执行层准入门禁说明

## 范围
本说明只定义执行层准入制度，不开启 execute，不改 workflow，不动 task-execute。

## 准入门禁

### 1) 隔离门禁
1. 单能力资源上限：CPU/内存/超时必须有明确上限。
2. 并发隔离：不得影响主程序核心接口可用性。
3. 错误隔离：失败不得传播为全局不可用。
4. 熔断策略：达到阈值自动熔断并保持可观测。

### 2) 回归门禁
1. 成功路径回归通过。
2. 失败路径回归通过。
3. 降级路径回归通过。
4. 冷启动与异常场景各至少一次回归记录。

### 3) 审计门禁
1. 状态变更、失败原因、回退动作必须有审计记录。
2. 运行状态可通过 `/api/plugins` 与 `/api/plugins/status` 查询。
3. 语义一致：`init_status/error_reason` 不得漂移。

### 4) 稳定性门禁
1. `GET /api/health` 正常。
2. `GET /api/plugins` 正常。
3. `GET /api/plugins/status` 正常。
4. `GET /api/vision/catalog` 正常。

### 5) 单刀验收门禁
1. 每次只允许一个 capability 申请进入执行层。
2. 通过后才能发起下一能力申请。
3. 申请失败必须先完成回退，再允许重提申请。

### 6) 回退门禁
1. 回退开关必须预先定义。
2. 回退触发条件必须量化（错误率/超时率/稳定性回归失败）。
3. 回退后必须恢复到“execute 关闭 + 管理层桥接可见”基线。

## 禁止清单
1. 禁止提前开启 `POST /api/plugins/:id/execute`。
2. 禁止改 `apps/local-api/src/workflow/index.ts`。
3. 禁止改 `apps/local-api/src/task-execute/*`。
4. 禁止把 planned_official 误判为可执行能力。

## 通过标准
同时满足 6 组门禁，且完成单刀验收与回退演练，才可讨论执行层灰度。
