## 一、推荐第一刀

### 🎯 切入口：**vision-bus catalog 插件化 + 视觉能力动态发现**

**即**：把 `vision-bus/index.ts` 中的硬编码 `VisionCatalog` 静态目录，改造为**插件发现的动态查询层**。

```
改造前（现状）：
  vision-bus/index.ts
    └─ getVisionCatalog()  →  读内存中的硬编码数组

改造后（目标）：
  vision-bus/index.ts
    ├─ 查 PluginManager 已注册的插件
    ├─ 按 capability=["vision"] 过滤
    └─ 动态合成 catalog（静态 + 动态合并）
```

**这一刀切的是 vision-bus，不是 workflow/index.ts**  
→ 不动13个executor  
→ 不动任何主流程  
→ 风险最小，收益最明确  

---

## 二、为什么先切这一刀

### 2.1 证据链

| 证据 | 内容 |
|------|------|
| E1 | vision-bus/index.ts 的 catalog 是全程序视觉管线的唯一发现入口 |
| E2 | catalog 已完整定义11个管线元数据+artifact类型+中间结果查询API |
| E3 | vision-bus 与 workflow 完全解耦（只读查询，无写依赖） |
| E4 | vision-bus 的 VisionCatalog 接口与 plugin-sdk 的 PluginManifest 接口高度同构（id/name/version/capabilities/status） |
| E5 | 插件激活后，PluginManager.listPlugins({capability:'vision'}) 可直接替代硬编码catalog |
| E6 | 这刀切完后，tracker/rule_engine 可作为第一批真实插件接入，无需再改 vision-bus |

### 2.2 风险/收益比

| | 评分 |
|---|------|
| 风险 | ⭐ 低（只改 vision-bus/index.ts，不动workflow） |
| 收益 | ⭐⭐⭐⭐ 高（插件发现层一次建成，视觉扩展从此可插拔） |
| 阻断依赖 | ⭐⭐ 低（PluginManager已是完整代码框架，只需激活+持久化） |
| 扩展性回报 | ⭐⭐⭐⭐⭐ 极高（后续 tracker/rule_engine/yolo变体 均沿此路径插件化） |

## 三、这一刀的具体施工范围

### Phase 1A（最小可用）：vision-bus → 插件动态发现

```
1. 创建插件DB表（最小集合）
   - plugins: id/plugin_id/name/version/capabilities/risk_level/enabled/status/registered_at
   - plugin_configs: plugin_id/config_key/config_value
   （参照 approvals 表结构，复用现有模式）

2. 改造 vision-bus/index.ts
   - getVisionCatalog() 增加：
     ├─ 读取 DB 中的已注册插件
     ├─ 按 capability 过滤 vision 相关能力
     └─ 合并硬编码静态catalog + 动态插件catalog
   - 新增 registerVisionPlugin(pluginId) → 插件注册时自动同步到 catalog
   - 新增 unregisterVisionPlugin(pluginId)

3. 激活 PluginManager
   - local-api server.ts 中 new PluginManager() 并持久化到DB
   - 插件注册/启用/禁用 → 写DB

4. demo-plugin 升级
   - plugins/builtin/demo-plugin/index.js 实现真实 execute（任意一个简单动作）
```

### Phase 1B（可选）：能力扩展

```
5. 扩展 plugin-sdk capabilities 枚举
   - 新增: vision / training / approval / deploy
   - 这使未来真正的视觉插件有标准能力标记

6. vision-bus catalog 暴露插件元数据
   - catalog.pipelines[].plugin_id（指向来源插件）
   - catalog.pipelines[].source（"builtin" | "plugin:xxx"）
```

## 四、主程序最容易乱的5个点（排序）

| 排名 | 模块 | 混乱潜力 | 说明 |
|------|------|----------|------|
| **#1** | workflow/index.ts（1500+行） | 极高 | 13个executor全塞一起，视觉硬编码，新增只能继续塞 |
| **#2** | classifier_verify executor（750+行） | 高 | 4个AND解冻条件+6个audit事件+真实推理+artifact写入+DB同步，往麻将叠加必乱 |

| **#5** | sam_handoff executor（400+行） | 中 | eval_result_summary_json反查dataset_yaml，路径推导脆弱 |

## 五、插件系统升级路线图

### 第一刀（当前）：vision-bus 插件化

**目标**：catalog 从硬编码 → 动态发现  
**改动范围**：vision-bus + PluginManager持久化 + 1张DB表  
**风险**：极低  
**工期估算**：小  

### 第二刀：tracker/rule_engine 插件化

**目标**：tracker_runner + rule_engine_runner 从mock升级为真实插件  
**前提**：第一刀完成（动态catalog已可用）  
**改动范围**：新增plugin包 + workflow路由桥接  
**理由**：tracker/rule_engine 是视觉链路的最后两环，当前mock状态影响pipeline完整性  

### 第三刀：workflow executor 插件化

**目标**：STEP_EXECUTORS 静态引用 → 动态分发  
**前提**：PluginManager持久化完整 + 插件API完善  
**风险**：高（workflow是主流程核心）  
**理由**：最后一步，等系统稳定后再动

## 六、绝对不能乱动的清单

| 禁止项 | 原因 |
|--------|------|
| ❌ 任何情况下不能直接改 workflow/index.ts 的 STEP_EXECUTORS | 13个executor的任意改动都可能破坏已有流程 |
| ❌ 不能在麻将T4完成前重构 classifier_verify | T4要接这条链路，必须保证稳定 |
| ❌ 不能先做 workflow桥接再做 vision-bus | catalog是发现层，顺序反了会导致tracker等插件无处注册 |
| ❌ 不能跳过DB持久化直接做热更新 | 无持久化的热更新是空谈 |
| ❌ 不能扩大 capabilities 枚举而不更新 manifest.ts | schema和SDK必须同步 |
| ❌ 不能在未激活 PluginManager 前写新插件 | 写了也无法运行 |
